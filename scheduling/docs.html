SYSTEM_KERNEL_MOD_01

First-Come
First-Served

Architecture: Linear FIFO Queue Mode: Non-Preemptive
Detailed Architectural Overview

First-Come, First-Served (FCFS) is the most fundamental CPU scheduling algorithm, operating on a strictly chronological basis. In this paradigm, the process that requests the CPU first is allocated the CPU first, managed through a standard First-In-First-Out (FIFO) queue. As a non-preemptive algorithm, once a process gains control of the CPU, it retains execution rights until it either terminates or initiates an I/O request. While conceptually simple, FCFS is highly sensitive to the arrival order and burst lengths. The most significant architectural weakness is the Convoy Effect: a phenomenon where several short processes are blocked by a single, massive CPU-bound process at the head of the queue, leading to poor device utilization and significantly inflated average waiting times.

Technical Advantages
  • Implementation Simplicity: Requires only a basic FIFO queue data structure, resulting in minimal development overhead.
  • Zero Preemption Overhead: No resources are wasted on frequent context switching or saving process states mid-execution.
  • Fairness in Order: Every process is treated equally based on its arrival time, ensuring no process is skipped or "jumped" in line.
  • Predictable Execution: For batch systems with known job lengths, the completion order is perfectly deterministic.
Technical Disadvantages
  • The Convoy Effect: Short, interactive tasks experience massive latency if stuck behind a long-running computation task.
  • High Average Waiting Time (AWT): Compared to SJF or Round Robin, the average time a process spends in the ready queue is usually much higher.
  • Poor Resource Utilization: I/O devices may sit idle while a CPU-bound process monopolizes the processor.
  • Non-Responsive: Entirely unsuitable for modern interactive operating systems where users expect immediate feedback.
9-Step Algorithmic Logic
01 Initialize the global System Clock (T) to zero and clear the Ready Queue.
02 Sort all incoming processes into a list based on their Arrival Time (AT) in ascending order.
03 Fetch the first process from the sorted list. If AT > T, advance the System Clock (T) to the Arrival Time (creating idle CPU time).
04 Calculate the Completion Time (CT): CT = T + Burst Time (BT).
05 Update the System Clock (T) to the current Completion Time (CT).
06 Calculate Turnaround Time (TAT): TAT = Completion Time (CT) - Arrival Time (AT).
07 Calculate Waiting Time (WT): WT = Turnaround Time (TAT) - Burst Time (BT).
08 Repeat steps 03 through 07 for every remaining process in the sorted list.
09 Compute the final System Performance metrics: Average TAT = ΣTAT / N and Average WT = ΣWT / N.

SYSTEM_KERNEL_MOD_02

Shortest Job
First (NP)

Architecture: Greedy Minimization Mode: Non-Preemptive
Detailed Architectural Overview

Shortest Job First (SJF) is a scheduling discipline that selects the process with the smallest execution burst from the set of available processes in the ready queue. This algorithm is theoretically optimal as it provides the **minimum average waiting time** for a fixed set of processes. By clearing shorter tasks quickly, it reduces the number of processes waiting in the queue, thereby maximizing system throughput. However, its primary architectural challenge is the requirement for "Future Knowledge"—the OS must know or predict the length of the next CPU burst. In non-preemptive SJF, once a process begins execution, it cannot be interrupted, even if a significantly shorter process arrives later, ensuring stability at the cost of immediate responsiveness to new arrivals.

Technical Advantages
  • Optimal Efficiency: Provably yields the lowest possible average waiting time for any set of non-preemptive tasks.
  • High Throughput: Processes are completed and removed from the system at the fastest possible rate.
  • Reduced Queue Congestion: Prevents the buildup of small tasks that often occurs in FCFS due to the convoy effect.
Technical Disadvantages
  • Starvation Risk: Longer processes may be delayed indefinitely if short-burst processes keep arriving (The "Indefinite Blocking" problem).
  • Estimation Complexity: Requires complex heuristics like Exponential Averaging to predict future burst times.
  • Unfairness: Prioritizes task length over arrival time, which may not be suitable for all user environments.
9-Step Algorithmic Selection Logic
01 Initialize the System Clock (T) and define a 'Completed' status flag for all processes.
02 Sort the master process list by Arrival Time (AT) to identify the initial entry point.
03 Scan the process list for all tasks where Arrival Time (AT) ≤ System Clock (T) and Status != 'Completed'.
04 From the filtered list, select the process with the **minimum Burst Time (BT)**.
05 In the event of a tie (two processes with equal BT), use Arrival Time (AT) as the tie-breaker (FCFS).
06 Update Completion Time (CT) = Current Clock (T) + Selected Process Burst Time (BT).
07 Set Status = 'Completed' for the selected process and advance System Clock (T) = CT.
08 Calculate TAT (CT - AT) and WT (TAT - BT) for the finished process.
09 Repeat from Step 03 until the number of completed processes equals the total process count.

SYSTEM_KERNEL_MOD_03

Priority
Scheduling (NP)

Architecture: Rank-Based Selection Mode: Non-Preemptive
Detailed Architectural Overview

Priority Scheduling is a sophisticated discipline where each process is assigned a numerical priority value. The CPU is allocated to the process with the highest priority (typically the lowest numerical value in most kernel implementations). In the non-preemptive variant, the scheduler evaluates the priority of all processes currently in the ready queue and selects the most "urgent" one to run to completion. This model is essential for systems that must distinguish between critical system-level tasks and background user applications. However, it introduces the significant risk of Starvation (or Indefinite Blocking), where low-priority processes may never execute if high-priority tasks are continuously injected into the system. This is traditionally mitigated using an "Aging" technique, where a process's priority is incrementally increased the longer it waits in the queue.

Technical Advantages
  • Service Differentiation: Allows the OS to prioritize mission-critical tasks over standard processing.
  • System Stability: Ensures that vital system daemons and drivers receive CPU cycles before user-level code.
  • User Control: Provides a mechanism for users or administrators to manually influence task execution order via "nice" values.
Technical Disadvantages
  • Indefinite Blocking: Low-priority processes can be completely starved of CPU time in a busy system.
  • Static Rigidity: Without dynamic aging, a process's importance remains fixed regardless of its wait time.
  • Overhead: Requires additional logic to manage priority queues and handle aging calculations.
9-Step Algorithmic Selection Logic
01 Initialize the System Clock (T) and the Priority Queue structure.
02 Assign a numerical Priority Value (P) to every process upon arrival.
03 At time T, identify all processes that have arrived (AT ≤ T) and are not yet completed.
04 Search the arrived processes for the one with the Highest Priority (Lowest Numerical Value).
05 If priorities are equal, use the FCFS (Arrival Time) rule as the secondary tie-breaker.
06 Dispatch the CPU to the selected process; it runs until the burst is fully exhausted (NP Mode).
07 Calculate CT = Current T + Burst Time, and advance the System Clock to CT.
08 Compute performance metrics: TAT = CT - AT and WT = TAT - BT.
09 Optionally apply the Aging Rule to increment the priority of all waiting processes before the next cycle.

SYSTEM_KERNEL_MOD_04

Multi-Level
Queue (NP)

Architecture: Partitioned Ready Queue Mode: Non-Preemptive Hierarchy
Detailed Architectural Overview

Multi-Level Queue (MLQ) scheduling is designed for scenarios where processes can be easily classified into different groups based on their nature or priority requirements. In this architecture, the single Ready Queue is split into several separate queues, such as System Processes, Interactive Processes, and Batch Processes. Each queue has its own priority level and can even utilize its own independent scheduling algorithm (e.g., Round Robin for interactive tasks and FCFS for batch tasks). In the non-preemptive version, the scheduler strictly follows a top-down approach: it will only pick a process from a lower-priority queue if all higher-priority queues are completely empty. Once a process starts, it remains in control of the CPU until completion, even if a task arrives in a higher-level queue during its execution.

Technical Advantages
  • Specialized Handling: Different types of processes receive treatment optimized for their specific behavior (I/O vs CPU bound).
  • Low Scheduling Overhead: Processes are permanently assigned to a queue, reducing the complexity of moving data between structures.
  • Strict Isolation: High-priority system tasks are physically separated from low-priority background user tasks.
Technical Disadvantages
  • Extreme Starvation: If the higher-level queues are constantly fed with new tasks, the lower-level queues may never execute.
  • Static Assignment: Processes cannot move between queues, leading to a lack of flexibility if a process changes behavior.
  • Inflexible Resource Allocation: A fixed percentage of CPU time cannot easily be guaranteed to lower queues without additional logic.

9-Step Algorithmic Selection Logic
01 Categorize every incoming process and assign it permanently to a specific Queue (Qn).
02 Initialize System Clock (T) and monitor all N sub-queues simultaneously.
03 Identify the highest-priority non-empty queue (e.g., Q0 is checked before Q1).
04 Select the head process from that queue using its specific internal algorithm (e.g., FCFS).
05 Execute the selected process; the CPU is held until the process terminates (NP Mode).

SYSTEM_KERNEL_MOD_05

Shortest Remaining
Time First

Architecture: Dynamic Greedy Preemption Mode: Preemptive
Detailed Architectural Overview

Shortest Remaining Time First (SRTF) is the preemptive counterpart to the SJF algorithm. In this model, the execution of a process can be interrupted if a new process arrives with a CPU burst length that is shorter than the remaining time of the currently executing process. The scheduler performs a comparison at every arrival event and every clock tick. This approach is highly efficient in terms of minimizing the Average Waiting Time (AWT) and Average Turnaround Time (TAT) because it ensures that the "quickest" tasks are always out of the system first. However, the architectural cost is significant: frequent preemption leads to increased context-switching overhead, as the system must constantly save and load Process Control Blocks (PCBs) whenever a "shorter" candidate enters the ready queue.

Technical Advantages
  • Optimal Performance: Provably the most efficient algorithm for minimizing average waiting time in a preemptive environment.
  • Superior Throughput: Small processes are completed with near-zero waiting time, clearing the queue rapidly.
  • Dynamic Adaptability: Responds instantly to new, short tasks, making it better for mixed workloads than non-preemptive SJF.
Technical Disadvantages
  • Context Switch Bloat: If many small processes arrive sequentially, the CPU may spend more time switching than executing.
  • Starvation: Long-running processes may never finish if the system is constantly saturated with shorter tasks.
  • Estimation Overhead: Like SJF, it requires accurate burst time predictions to function effectively.
9-Step Algorithmic Selection Logic
01 Maintain a table of Remaining Burst Times (RT) for all arrived processes, initialized to their total BT.
02 At each unit of time (T), identify all processes that have arrived (AT ≤ T) and are not yet finished.
03 Select the process with the minimum Remaining Time (RT) from the available pool.
04 If a new process arrives with RT < current process RT, trigger an immediate preemption and context switch.
05 Decrement the RT of the currently executing process by 1 at every clock pulse.
06 If a process's RT reaches 0, mark it as 'Finished' and record the current T as its Completion Time (CT).
07 Update System Clock (T = T + 1) and repeat the selection logic.
08 Calculate TAT = CT - AT and WT = TAT - BT (using the original Burst Time).
09 Aggregate all individual stats to determine global Mean Waiting Time and Mean Turnaround Time.

SYSTEM_KERNEL_MOD_06

Priority
Scheduling (P)

Architecture: Immediate Interrupt Mode: Preemptive
Detailed Architectural Overview

Preemptive Priority Scheduling is a real-time scheduling discipline where the CPU is always allocated to the process with the highest priority currently in the system. Unlike the non-preemptive version, if a process arrives at the ready queue with a priority higher than the currently running process, the kernel performs an immediate Context Switch. The running process is suspended, its state is saved back to its Process Control Block (PCB), and the CPU is granted to the new arrival. This ensures that critical system interrupts or high-priority user demands are serviced with the lowest possible latency. While this provides maximum responsiveness for "VIP" tasks, it significantly increases the complexity of the scheduler and makes the "Starvation" of low-priority tasks a constant operational risk.

Technical Advantages
  • Real-Time Responsiveness: High-priority tasks are guaranteed the CPU the microsecond they arrive.
  • Urgency Handling: Ideal for embedded systems or kernels where certain events (like hardware failures) must be handled instantly.
  • Granular Control: Provides total control over task execution order based on importance rather than time.
Technical Disadvantages
  • High Overhead: Constant priority comparisons and potential context switches at every arrival event consume CPU cycles.
  • Extreme Starvation: Low-priority tasks can remain in the 'Ready' state indefinitely if high-priority traffic is high.
  • Data Inconsistency: Preempting a process mid-calculation requires robust synchronization to prevent memory corruption.
9-Step Algorithmic Logic
01 Maintain a sorted list of all active processes based on their Priority Value (P).
02 At each clock cycle (T), check the priority of the arriving process against the current process.
03 If New_P < Current_P (where lower is higher priority), save the current process state and preempt.
04 Update the Remaining Burst Time (RT) of the preempted process.
05 Execute the highest-priority task for one time unit or until a higher-priority task arrives.
06 When a process completes (RT=0), record the current T as the Completion Time (CT).
07 Select the next highest priority process from the ready queue to resume or start.
08 Apply the Aging formula: P = P - 1 for every X units a process spends waiting in the queue.
09 Calculate final stats: TAT = CT - AT and WT = TAT - Original_BT.

SYSTEM_KERNEL_MOD_07

Round Robin
Scheduling

Architecture: Time-Slicing Cyclic Queue Mode: Preemptive (Clock-Driven)
Detailed Architectural Overview

Round Robin (RR) is the foundational scheduling algorithm for all modern time-sharing operating systems. It defines a rigid unit of time called the Time Quantum (δ) or Time Slice. The CPU scheduler treats the ready queue as a circular FIFO structure, granting each process a maximum of one Time Quantum. If a process's burst exceeds the quantum, it is forcibly preempted, its context is saved, and it is appended to the tail of the queue. This ensures that every process receives a fair share of the CPU and provides a guaranteed upper bound on response time, making it ideal for user-facing environments where interactivity is key. The efficiency of RR is heavily dependent on the size of the Time Quantum: if too large, it degrades into FCFS; if too small, the system collapses under context-switching overhead.

Technical Advantages
  • Maximum Fairness: Every process is guaranteed a CPU slice in every (N-1)δ time units, preventing any single task from monopolizing the processor.
  • Superior Responsiveness: Excellent for interactive applications where the user expects immediate feedback from multiple open programs.
  • Starvation-Free: By design, no process is ever blocked indefinitely regardless of its total burst size.
Technical Disadvantages
  • Context Overhead: Frequent preemption leads to high context-switching latency, which can consume significant CPU cycles.
  • Quantum Sensitivity: Finding the "Sweet Spot" for the Time Quantum is difficult and often requires empirical tuning.
  • High Average Waiting Time: Because processes are constantly interrupted, the AWT is typically higher than in SJF or SRTF.
9-Step Algorithmic Logic
01 Define a fixed Time Quantum (δ) and initialize an empty FIFO Ready Queue.
02 Push all processes arriving at T=0 into the Ready Queue in the order of their arrival.
03 Pop the first process (P_curr) from the head of the queue and execute for time = min(Remaining_Burst, δ).
04 Continuously monitor for any new process arrivals during the execution of P_curr.
05 Upon arrival, immediately push new processes to the tail of the Ready Queue.
06 If P_curr finishes its burst within the quantum, calculate its Completion Time (CT) immediately.
07 If P_curr exceeds the quantum, preempt it and push it to the tail of the queue *after* any new arrivals from Step 05.
08 Repeat the process until the Ready Queue is null and no more processes are expected.
09 Aggregate TAT (CT - AT) and WT (TAT - BT) to assess total system efficiency.

SYSTEM_KERNEL_MOD_08

Multi-Level
Queue (P)

Architecture: Tiered Interrupt Engine Mode: Strict Preemption
Detailed Architectural Overview & Preemption Logic

The Preemptive Multi-Level Queue (MLQ) is a sophisticated kernel-level scheduling architecture where the Ready Queue is partitioned into multiple distinct sub-queues based on process priority or type. Each queue operates with a strict dominance hierarchy (e.g., Q0 > Q1 > Q2).

In the Preemptive variant, the scheduler enforces immediate CPU reclamation. If the processor is currently executing a task from a lower-priority tier (like a background batch job in Q2) and a high-priority task arrives in a superior tier (like a system interrupt in Q0), the kernel triggers an immediate Context Switch. The state of the lower-priority process is saved into its Process Control Block (PCB), and the CPU is instantly dispatched to the higher-tier arrival. This architecture ensures near-zero latency for mission-critical tasks but requires careful management to prevent lower tiers from complete starvation.

Technical Advantages
  • Guaranteed Response: High-priority system tasks are never delayed by user-level processes.
  • Hardware Isolation: Effectively separates time-critical I/O handling from standard computation.
  • Organizational Clarity: Allows different scheduling policies to be applied to different types of tasks (e.g., RR for interactive, FCFS for batch).
Technical Disadvantages
  • Static Rigidity: Processes are permanently tied to a queue; they cannot adapt if their behavior changes.
  • Starvation Risk: Lower-tier processes can be indefinitely blocked if higher-priority queues are constantly busy.
  • Interrupt Overhead: Frequent preemptions between tiers can degrade overall system throughput due to context-switch costs.

SYSTEM_KERNEL_MOD_09

Multi-Level
Feedback Queue

Architecture: Adaptive Migration Engine Mode: Dynamic Feedback Loop
Detailed Architectural Overview & Adaptive Feedback

Multi-Level Feedback Queue (MLFQ) is the industry-standard scheduling algorithm for general-purpose OS kernels (NT, Unix-like). Unlike standard MLQ, it does not rely on static process classification. Instead, it learns and adapts based on the execution profile of each task.

The core philosophy is simple: favor short, interactive, or I/O-bound tasks and penalize long-running, CPU-intensive background tasks. If a process exhausts its full time quantum in a high-priority queue (Q0), the scheduler "punishes" it by demoting it to a lower-priority queue (Q1). If it exhausts the Q1 quantum, it is demoted to the FCFS queue (Q2). This allows the system to achieve the low response time of "Shortest Job First" without requiring any prior knowledge of the process's burst time. To prevent starvation of demoted tasks, a periodic Priority Boost promotes all tasks back to Q0, ensuring every process eventually gets CPU time.

Technical Advantages
  • Optimal Responsiveness: Interactive processes finish in the top tier with almost zero wait time.
  • Dynamic Flexibility: Processes can change their behavior (from CPU-bound to I/O-bound) and the scheduler will adjust accordingly.
  • Auto-Balancing: Naturally balances throughput and latency without manual administrative tuning.
Technical Disadvantages
  • High Complexity: Implementing the migration logic and boosting timers is extremely complex in a kernel.
  • Context Switching: Frequent demotions and pre-emptions increase the overall system overhead.
  • Parameter Sensitivity: System performance depends heavily on the chosen TQ values and the boost interval.
