UMA & NUMA
Uniform vs Non-Uniform Memory Access
How processors share memory fundamentally shapes system performance. UMA gives every CPU identical access latency to all memory. NUMA partitions memory into nodes — local access is fast, remote is slower. Click any CPU, then any memory block to measure the exact latency for that pair.
Uniform Memory Access
Symmetric Multiprocessing

In a UMA (Uniform Memory Access) architecture, all processors share a single physical memory space through a common interconnect. The defining characteristic is that the memory access latency is identical for every CPU, regardless of which memory location is accessed.

The two classic UMA topologies are the shared bus (all CPUs and memory on one electrical bus — simple but creates a bottleneck as CPUs scale) and the crossbar switch / memory interconnect (a switching fabric that allows simultaneous non-conflicting memory accesses). Both guarantee uniform latency but differ in bandwidth and scalability. UMA is the model used in most commodity desktop/laptop multi-core systems.

Non-Uniform Memory Access
Multi-Node Architecture

In a NUMA (Non-Uniform Memory Access) architecture, memory is physically distributed across multiple nodes, each containing one or more CPUs and a local memory bank. A CPU accessing its local memory experiences the lowest latency. Accessing memory on a remote node requires traversing one or more inter-node interconnects (e.g. AMD Infinity Fabric, Intel QPI/UPI), incurring additional hop latency at each crossing.

The NUMA distance (or hop count) between two nodes determines the penalty. Modern Linux and Windows are NUMA-aware — the scheduler tries to keep threads on the same node as their data. Poor NUMA affinity is a common source of hidden performance bottlenecks in high-core-count server systems (e.g. AMD EPYC, Intel Xeon Scalable, AWS Graviton). The NUMA factor (ratio of remote to local latency) typically ranges from 1.5× to 3×.

NUMA-awareness in practice: Operating systems expose NUMA topology via interfaces like /sys/devices/system/node/ on Linux. The numactl tool lets administrators pin processes and memory to specific nodes. High-performance databases (PostgreSQL, Redis) and runtimes (JVM) have NUMA-aware memory allocators that exploit local access patterns. Ignoring NUMA in a 64-core server can silently reduce throughput by 30–60%.
